Setting Up an LLM Provider

Capella Agent supports 10 LLM providers. You need at least one configured before the agent can respond to queries.

Recommended for first-time users: GitHub Models is completely free (no billing required), supports GPT-4o and several other models, and requires only a GitHub personal access token which you likely already have.

Opening the Preferences Page

All provider configuration lives in Window → Preferences → Capella Agent → LLM Providers.

Select a provider from the left-hand list, enter the required fields, and click Test Connection. A green checkmark confirms the key works. Click Apply and Close.

Claude (Anthropic)

  1. Go to console.anthropic.com and sign up or log in.
  2. Navigate to API Keys and create a new key.
  3. Copy the key (it is shown only once).
  4. In Capella Preferences, select Claude (Anthropic).
  5. Paste the key into the API Key field.
  6. Select a model — claude-opus-4-6 is the most capable, claude-haiku-4-5-20251001 is fastest, claude-sonnet-4-6 is the balanced default.
  7. Click Test Connection.
Saved keys are never displayed. After you click Apply, Capella Agent stores the key in Eclipse secure preferences and replaces the field with a •••••• (saved — leave blank to keep) placeholder. To keep your current key, leave the field blank and make your other changes. To replace it, type a new key and click Apply. The key never leaves your machine except in outbound requests to the provider.
Best model for MBSE: Claude excels at structured reasoning and long-context analysis, making it ideal for large Capella models with hundreds of elements.

OpenAI

  1. Go to platform.openai.com/api-keys.
  2. Create a new secret key and copy it.
  3. In Capella Preferences, select OpenAI.
  4. Paste the key and select a model (gpt-4o recommended).
  5. Click Test Connection.

GitHub Models (Free)

  1. Go to github.com/settings/tokens.
  2. Generate a new token (classic). No special scopes are required — a token with no selected scopes works for GitHub Models.
  3. Copy the token.
  4. In Capella Preferences, select GitHub Models.
  5. Paste the token into the API Key field.
  6. Select a model — gpt-4o or Meta-Llama-3.1-405B-Instruct.
  7. Click Test Connection.

Groq (Free)

  1. Go to console.groq.com and sign up.
  2. Create an API key.
  3. In Capella Preferences, select Groq.
  4. Paste the key and select llama-3.3-70b-versatile or mixtral-8x7b-32768.
  5. Click Test Connection.
Performance note: Groq uses custom inference hardware and is extremely fast — often 10-20x faster than OpenAI for the same model size. Ideal for interactive queries.

Ollama (Offline / Air-Gapped)

  1. Install Ollama from ollama.com on your local machine.
  2. Pull a model: ollama pull llama3.1 (or mistral, codellama, etc.).
  3. Start Ollama: ollama serve (it runs on http://localhost:11434 by default).
  4. In Capella Preferences, select Ollama.
  5. The base URL defaults to http://localhost:11434 — change this if Ollama is on another host.
  6. No API key is required.
  7. Select the model you pulled from the dropdown (Capella Agent queries Ollama for the list).
  8. Click Test Connection.
ITAR / air-gapped: When using Ollama or LM Studio, no data leaves your machine. This is the approved path for ITAR-controlled program models.

Cost Controls (Safety Baselines)

Every provider configuration in Capella Agent runs under a fixed set of cost and rate guardrails. These are deliberate v1.0 safety baselines, not placeholders — they protect you from a runaway tool-calling loop, a compromised session, or a provider outage that would otherwise burn credit silently.

GuardrailDefaultWhat it protects against
Per-session token budget1,000,000 tokensRunaway loops inside a single chat session
Per-day hard cap (UTC)10,000,000 tokensCompromised workstation, forgotten background session
Provider circuit breaker5 errors in 120 s → open 5 minSilent retries during a provider outage
Iteration hard ceiling50 tool calls per turnTool-call ping-pong between the model and the workspace
Tool-result byte cap200 KB per toolA single huge result (e.g. list_all_requirements on a 5000-row project) blowing the budget
Why fixed, not a preference? These numbers are calibrated against real-world provider pricing and outage patterns. A budget that a user can raise with one click is a budget that protects nothing. Administrators who need a different cap can call DailyTokenLimiter.getInstance().setCap(long) from a startup hook — see the Security & Governance page for details.
