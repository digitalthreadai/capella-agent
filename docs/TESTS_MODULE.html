The Tests Module

What com.capellaagent.core.tests actually is, who runs it, and why it lives in its own bundle instead of src/test/java.

The one-line answer

com.capellaagent.core.tests is a developer-only artifact. End users never see it, never install it, never run it. It exists for one job: prove the production code in com.capellaagent.core does what we say it does.

Who runs it, and when

You, manually

After any code change, before pushing.
mvn verify from repo root.

CI, automatically

Every git push when CI is wired (GitHub Actions).

End users

Never. The test fragment is not in feature.xml, so the p2 install does not ship it.

What it actually is

It's a Tycho eclipse-test-plugin (not an eclipse-plugin). That packaging is special: when Maven/Tycho hits it during mvn verify, it spins up a headless Eclipse OSGi container, loads the test fragment alongside com.capellaagent.core, and runs JUnit 5 tests against the real OSGi-resolved classpath.

That matters because most of our code touches Eclipse APIs — TransactionalEditingDomain, IExtensionRegistry, IPreferenceStore — and those only behave correctly when an OSGi container is wired up around them. A plain JVM unit test would fail to resolve half the classes.

What's inside

167 unit tests across 12 test classes covering things like:

  • TokenEstimator — does the char/4 heuristic stay within 10% on real prompts?
  • HistoryWindow — does it correctly evict orphaned tool_result messages so Groq does not 400?
  • ToolResultSanitizer — does the prompt-injection wrap/detect/gate pattern actually catch payloads?
  • WriteToolGate — does it correctly block destructive tools in read-only mode?
  • JsonFileConversationStore — does session crash recovery survive a kill -9?
  • ChatSessionController — does the tool-call loop terminate? Does it handle 6 retries without infinite-looping?
  • WelcomeWizardModel — do all state transitions in the (deferred) wizard data layer behave?

How to run them

# All tests across every module (the production way)
mvn clean verify

# Just the core test bundle
mvn -pl com.capellaagent.core.tests -am verify

# Skip tests during a packaging-only build
mvn clean package -DskipTests

Tycho's surefire plugin reports go to com.capellaagent.core.tests/target/surefire-reports/. A green run looks like: 

[INFO] Tests run: 167, Failures: 0, Errors: 0, Skipped: 0
[INFO] BUILD SUCCESS

Why a separate module instead of src/test/java?

Standard Maven puts tests next to production code in src/test/java. We can't, for three independent reasons:

ReasonWhy it forces a separate bundle
Tycho packaging eclipse-plugin builds one OSGi bundle, period. To get tests, you need a second bundle — that's what the .tests fragment is.
OSGi visibility The test bundle is declared as a fragment of com.capellaagent.core. Fragments share the host's classloader, which means tests can see package-private classes without us having to bump everything to public. (You may remember the formatError / removeOrphans “Public for testing” javadoc — those were the few places we couldn't avoid it.)
Headless OSGi runtime Tycho's surefire plugin needs the test fragment to know how to bootstrap a real Eclipse runtime around the tests. A plain src/test/java folder cannot do that — it would fail the moment any test loads org.eclipse.emf.transaction.*.

What you'd lose without it

  • No safety net when refactoring ChatSessionController, HistoryWindow, or TokenEstimator.
  • No way to verify the prompt-injection defense actually fires before shipping a release.
  • No regression coverage for the Groq orphaned-tool_result bug that took a day to track down.
  • The smoke test checklist becomes the only QA — and a 30-minute manual loop cannot cover 167 edge cases.

How it relates to the smoke test

The smoke test checklist and the tests module are complementary, not redundant:
